# Legacy Code Analyzer - 分析工作流详细指南

## 语言一致性规则（最高优先级）

**CRITICAL**: 所有分析报告、代码注释和回复必须与用户输入使用相同语言。
- 检测 `<user_input>` 的语言并使用该语言生成所有输出
- 如果用户输入是中文，则生成中文分析报告
- 如果用户输入是英文，则生成英文分析报告
- 此规则优先级高于任何英文模板或示例

## 分析前准备

### 1. 确认分析范围
在开始任何分析前，你必须使用 `AskUserQuestion` 确认分析范围：

```
问题：请选择分析范围
选项：
  - 完整分析（所有8个模块）[推荐]
  - 仅结构与元数据（模块1）
  - 语义解析与设计意图（模块2）
  - 依赖与耦合度分析（模块3）
  - 代码质量评估（模块4）
  - 风险预警与修改指导（模块5）
  - 测试与重构辅助（模块6）
  - 目标性评估（模块8，需提供需求文档）
```

如果用户已明确指定范围，跳过此步骤。

### 2. 检查需求文档
如果用户提供了需求文档内容或路径，必须先完整读取并解析后再开始分析。
需求文档可以是：
- 纯文本描述
- Markdown 格式的需求列表
- 用户故事格式
- 验收标准列表
- 文件路径指向需求文档

### 3. 使用 TodoWrite 规划分析步骤
将分析任务分解为可追踪的子任务，每完成一个模块就标记为完成。

## 各模块执行细则

### 模块 1 执行细则：代码扫描

#### Step 1.1 - 文件发现
```bash
# 使用 Glob 发现所有源代码文件
Glob: pattern="**/*.{py,java,js,jsx,ts,tsx,cpp,h,hpp,c,cc,cxx}"
# 同时发现配置文件
Glob: pattern="**/{package.json,pom.xml,build.gradle,CMakeLists.txt,requirements.txt,setup.py}"
```

#### Step 1.2 - 语言识别
按以下优先级识别语言：
1. 配置文件（package.json → JavaScript/TypeScript）
2. 文件扩展名（.py → Python, .java → Java, .cpp/.h → C++）
3. 代码内容特征（语法关键词匹配）

#### Step 1.3 - 规模统计
使用 `Grep` 统计：
- 总行数：`wc -l`
- 函数/方法定义：`Grep: pattern="def |function |public.*\(|void.*\("`
- 类定义：`Grep: pattern="class "`
- 注释行：`Grep: pattern="^\s*(#|//|/\*|\*|<!--)"`

#### Step 1.4 - 技术栈识别
读取配置文件提取：
- 框架依赖
- 数据库驱动
- 测试框架
- 构建工具
- 运行时版本要求

### 模块 2 执行细则：语义解析

#### Step 2.1 - 逐文件读取
对每个源文件：
1. 使用 `Read` 读取完整文件内容
2. 如果文件超过 2000 行，分批读取
3. 重点标记包含 `main`、`init`、`run`、`start` 等入口函数的文件

#### Step 2.2 - 控制流分析
手动分析每个函数的控制流：
1. 识别所有条件分支（if/else/switch/match）
2. 识别所有循环（for/while/do-while）
3. 识别所有异常处理（try/catch/except/finally）
4. 识别所有跳转（return/break/continue/goto）
5. 标记不可达代码

#### Step 2.3 - 设计模式识别
查找以下模式特征：
- **Singleton**: 私有构造函数 + 静态实例
- **Factory**: 返回不同子类实例的创建方法
- **Observer**: 注册/通知回调模式
- **Strategy**: 接口 + 多个可替换实现
- **Decorator**: 包装类增强功能
- **Chain of Responsibility**: 链式处理请求

#### Step 2.4 - 临时方案标记
搜索以下模式：
```bash
Grep: pattern="TODO|FIXME|HACK|XXX|TEMP|WORKAROUND|hotfix|quick.?fix"
Grep: pattern="^\s*//.*\b(?:hack|temp|fixme)\b"
```

### 模块 3 执行细则：依赖分析

#### Step 3.1 - 显式依赖提取
```bash
# Python imports
Grep: pattern="^(import |from )"
# Java imports
Grep: pattern="^import "
# JavaScript imports
Grep: pattern="^(import |const .*= require|export )"
# C++ includes
Grep: pattern="^#include"
```

#### Step 3.2 - 隐式依赖检测
搜索以下模式：
```bash
# 全局变量
Grep: pattern="^(global |window\.|GLOBAL_|global_|_global)"
# 环境变量
Grep: pattern="(process\.env|os\.environ|System\.getenv|getenv)"
# 文件系统操作
Grep: pattern="(open\(|fs\.|File\(|fstream)"
# 反射/eval
Grep: pattern="(eval\(|exec\(|reflect|Class\.forName)"
```

#### Step 3.3 - 耦合度计算
手动构建模块依赖矩阵：
1. 列出所有模块
2. 对每对模块 (A, B)，检查 A 是否依赖 B
3. 计算 Ca（多少模块依赖本模块）和 Ce（本模块依赖多少模块）
4. 计算 I = Ce / (Ca + Ce)

#### Step 3.4 - 循环检测
使用 DFS 算法：
1. 从每个模块出发进行深度优先搜索
2. 维护访问栈
3. 如果遇到已在栈中的模块 → 发现循环
4. 记录完整循环路径

### 模块 4 执行细则：质量评估

#### Step 4.1 - 圈复杂度计算
对每个函数，按以下规则计数：
- 基础复杂度 = 1
- 每个 if/elif/else +1
- 每个 for/while +1
- 每个 case +1
- 每个 and/or（短路运算符）+1
- 每个 except/catch +1

#### Step 4.2 - 可维护性指数计算
MI = 171 - 5.2 × ln(Halstead Volume) - 0.23 × CC - 16.2 × ln(LOC)
简化版：
- V（Halstead Volume）≈ 操作符数 + 操作数
- CC = 圈复杂度
- LOC = 有效代码行数

#### Step 4.3 - 安全漏洞检测清单
对照以下检查清单逐项检查：

**注入检测**：
```bash
# SQL注入风险
Grep: pattern="(\+.*sql|f\".*select|format.*query|\.execute\(.*\+)"
# 命令注入风险
Grep: pattern="(os\.system|subprocess\.call|exec\(|Runtime\.exec)"
```

**XSS检测（JavaScript）**：
```bash
Grep: pattern="(innerHTML|document\.write|dangerouslySetInnerHTML)"
```

**敏感数据暴露**：
```bash
Grep: pattern="(password|secret|api_key|token|private_key)\s*="
```

**缓冲区溢出（C/C++）**：
```bash
Grep: pattern="(strcpy|strcat|sprintf|gets|scanf)"
```

### 模块 5 执行细则：风险预警

#### Step 5.1 - 风险分类
按以下标准对每个缺陷分类：

| 等级 | 标准 |
|------|------|
| 🔴 高 | 可导致安全漏洞、数据损坏、系统崩溃 |
| 🟡 中 | 影响可维护性、可能引发运行时错误 |
| 🟢 低 | 代码异味、风格问题、潜在优化点 |

#### Step 5.2 - 连锁影响分析
对高风险代码：
1. 使用模块 3 的依赖图找出所有依赖该代码的模块
2. 分析修改后每个依赖模块需要做的调整
3. 标注影响范围（局部/模块级/系统级）

### 模块 6 执行细则：测试与重构

#### Step 6.1 - 测试用例生成规则
对每个核心函数：
1. **正常场景**：至少 2 个典型输入
2. **边界场景**：空值、零值、最大值、最小值
3. **异常场景**：无效输入、类型错误、越界访问

生成的测试用例必须：
- 使用项目已有的测试框架
- 覆盖函数的所有条件分支
- 包含清晰的断言说明

#### Step 6.2 - 重构方案生成规则
当函数满足以下任一条件时，生成重构方案：
- 圈复杂度 > 15
- 函数行数 > 50
- 可维护性指数 < 50
- 重复代码块 > 2 处

重构方案必须：
- 保持原有功能不变
- 逐步拆分，每步可独立验证
- 提供重构前后指标对比

### 模块 8 执行细则：需求覆盖分析

#### Step 8.1 - 需求解析
从用户提供的需求文本中自动提取：
1. 按编号识别：`FR-\d+`, `NFR-\d+`, `BR-\d+`, `CN-\d+`
2. 按关键词识别：`需求|Requirement|User Story|验收|Acceptance`
3. 按格式识别：列表项、编号列表、表格行

#### Step 8.2 - 需求对齐算法
对每个需求项：
1. 提取关键词（名词短语、动词短语）
2. 在代码库中搜索匹配：
   - 函数名匹配
   - 注释匹配
   - 业务逻辑语义匹配
3. 如果找到匹配 → 判定实现程度
4. 如果未找到 → 尝试推断"暗含实现"

#### Step 8.3 - 暗含实现检测
暗含实现的判定条件：
- 代码实现了某功能，但函数名/注释中未提及对应需求
- 功能逻辑完整但需求文档未显式覆盖
- 标记为 🔮 并在报告中说明推断依据

## 报告质量标准

### 自检清单
在输出最终报告前，确认以下内容：
- [ ] 所有代码引用包含精确的文件路径和行号
- [ ] 所有 Mermaid 图节点标签使用引号包裹
- [ ] 风险等级使用统一的 🔴🟡🟢 标识
- [ ] 需求覆盖矩阵包含所有提取的需求项
- [ ] 每个缺陷都附有修复建议
- [ ] 报告语言与用户输入语言一致
- [ ] 所有表格对齐正确
- [ ] 依赖分析包含显式和隐式依赖

### 禁止事项
- ❌ 不要在没有读取文件的情况下推测代码内容
- ❌ 不要忽略隐式依赖（全局变量、环境变量）
- ❌ 不要跳过安全漏洞检测
- ❌ 不要生成不完整的需求覆盖矩阵
- ❌ 不要在风险预警中模糊描述（必须标注精确位置）

## 交互式探索模式

当用户进行交互式查询时（模块 7），遵循以下规则：

### 查询响应模板

**功能解释类**：
```
[功能名称] 位于 [文件路径:L行号]

**功能概述**：[1-2句话描述]

**核心逻辑**：
[Mermaid flowchart 流程图]

**关键变量**：
| 变量 | 类型 | 作用 |
|------|------|------|
| ... | ... | ... |
```

**设计意图类**：
```
基于代码分析，推断该设计的原始意图为：

1. **[意图1]**：[代码证据]
2. **[意图2]**：[代码证据]

**可能的设计约束**：
- [约束1]
- [约束2]

**⚠️ 设计问题**：
- [问题1]：[说明]
```

**依赖查询类**：
```

- [调用者1]：[文件:L行号]

**风险等级**：🔴/🟡/🟢
```
